# OmniEdge (全域工联)基础设施规划书
## Phase 2：规模化运营期 — 3,000 家商户 / 多品类扩展

---

## 一、阶段目标

在 Phase 1 SOP 验证成功的基础上，Phase 2 进入规模化运营：

- 商户规模：从 POC 扩展到 **3,000 家活跃商户**
- 品类扩展：从首个垂直品类拓展到 **3-5 个品类**（电子元件、AI 硬件、手机配件等）
- 覆盖市场：越南、泰国、新加坡、马来西亚四国
- 日均撮合量：**10,000-15,000 笔**
- 系统要求：高可用双机热备 + 多机 AI 并行推理 + 99.5% SLA

---

## 二、系统架构

Phase 2 升级为 **双机云端负载均衡 + 2 节点本地 AI 集群 + 云端 API** 的混合架构：

- **香港云端（2 台负载均衡）**：双机热备，任一台宕机自动切换
- **本地 AI 推理集群（2 台工作站）**：分工协作，互为降级备份
- **云端 API**：DeepSeek 持续承担寻源辅助，OpenAI 处理多语言生成
- **加密隧道**：WireGuard VPN + 企业专线，保障稳定性

从 Phase 1 到 Phase 2 的关键升级：
1. 云端从单台升级为 **双机负载均衡**，消除单点故障
2. 本地从单台升级为 **2 台 GPU 工作站**，支撑多品类并发推理
3. 数据库从 Serverless 升级为 **固定规格实例**，保障稳定吞吐
4. 新增 **企业光纤专线**，替代千兆宽带，VPN 延迟更稳定

## 四、本地 AI 推理集群（2 台工作站）

### 4.1 为什么升级到 2 台

3,000 家商户 + 多品类带来的并发压力：

| 模块 | AI 负载 | 显存需求 |
|------|---------|---------|
| 询盘解析 + 谈判策略 | qwen3:14b 持续推理 | 10GB |
| 供应商寻源 (LLM Sourcing) | qwen3:14b 第二实例 | 10GB |
| 视觉自愈 (VLM) | qwen-vl:7b 图像识别 | 5GB |
| 交易员辅助查询 | 实时自然语言查询 | 共享模型 |
| RPA 数据提取 | 结构化信息抽取 | 共享模型 |

单台 RTX 4090（24GB）无法同时满足多品类并发推理。2 台工作站按职责分工，互为降级备份。

## 九、风险控制

| 风险 | 应对措施 |
|------|---------|
| 云服务器故障 | 双机负载均衡，单台宕机自动切换 |
| 数据库故障 | 托管服务自动每日备份 + 7 天回溯 |
| 本地工作站故障 | 2 台互为备份，任一台故障另一台接管 |
| 本地断电 | 每台配备 UPS，支撑 15-30 分钟安全关机 |
| VPN 隧道中断 | 企业专线稳定性 + 自动降级到外部 API |
| DDoS 攻击 | Cloudflare 企业级防护 |
| 数据泄露 | 全链路 AES-256 加密 + VPC 隔离 + 硬件绑定 |
